# SEGameTycoon — Mod 安全执行环境（沙箱系统）功能规格提案

版本：v1.0
目标：在保持游戏绝对开放性的前提下，为 GDScript 和 C# Mod 提供基于底层 API 劫持的、与操作系统同级的权限控制与安全隔离。

---

## 一、 核心设计哲学：从"静态扫描"转向"API 劫持"

当前 ModSecurityScanner 依赖静态文本匹配，无法防御动态生成（load、new）、字符串拼接或底层绕行。新版沙箱系统将放弃"防代码"，转向 "防执行与路径拦截"。

实现方式：
沙箱主类采用阻塞式同步加载，优先级最高，在所有 Mod 初始化之前完成对引擎底层 API 的劫持。

- 利用 C# 的 NativeInterop 或 .NET 的 P/Invoke 绑定，直接截获（Hook）Godot 引擎原生 C++ 层的函数表（VTables）（如 FileAccess::open、DirAccess::list_dir）。
- 效果：无论 GDScript 还是 C# Mod，所有涉及文件操作的调用，底层实际执行的将是沙箱的校验与重定向函数。
- 优点：无需修改 GDScript 语法（无需禁用 call 或 new）。Mod 逻辑完全不受影响，数据流向已被沙箱完全接管。

---

## 二、 文件系统隔离与权限申请 (File System Virtualization)

采用 "基于路径哈希表的文件系统虚拟化" 机制，取代笼统的"沙箱终端"描述。

### 1. 严格/绝对严格模式（隔离区）

- 底层逻辑：为每个 Mod 创建专属的物理沙箱目录（`user://mods_sandbox/{ModID}/`）。
- 拦截规则：拦截底层文件 API。一旦 Mod 企图访问 `user://`、`res://` 或任何绝对路径（如 `C:/Users/`），沙箱将路径自动重定向（Redirect）至 Mod 的专属沙箱目录。
- 结果：Mod 认为自己成功读写，实际上仅触碰了自身的隔离区，游戏本体与其他 Mod 数据绝对安全。

### 2. 文件权限申请（通过 ModAPI 暴露）

- Method Definition：在 ModBridge.cs 中新增公开方法，供沙箱调用：

```csharp
public static bool RequestFilePermission(string modId, string targetPath, PermissionType type) { ... }
public enum PermissionType { Read, Write, ReadWrite }
```

- UI 联动：沙箱底层拦截器捕获违规操作后，触发 `ModBridge.RequestFilePermission`。该方法会在游戏主 UI 层（UiLayer）弹出模态面板弹窗："Mod [名称] 想要 [读写] 路径 [xxx]，是否允许？"。
- 持久化存储：玩家选择记录至 `user://mods_permissions.json`，下次运行自动通过。

### 3. 自定义模式（玩家定义白名单）

- 在游戏设置菜单中增加"Mod 权限管理"面板。
- 玩家可手动编辑准入的绝对路径白名单（如允许 `C:/Users/你的名字/Downloads/` 读写）。
- 所有规则采用 `Dictionary<string, int>` 哈希表存储（键为路径，固定值为 0，存在即允许）。

## 三、 C# 与 GDScript 的底层拦截对齐（防御统一）

确保 GDScript 防御力度与 C# 完全一致，消除弱点。

### 1. 反射 (Reflection) 与 unsafe 代码拦截（核心防线）

- C#：
  - 反射：在 ModAssemblyLoader 中使用 .NET AssemblyLoadContext 加载时，禁止解析 System.Reflection 和 System.Runtime.CompilerServices.Unsafe 命名空间调用，阻止绕过 API。
  - unsafe 致命打击：unsafe 允许代码直接操作内存指针，绕过托管边界的拦截。在元数据加载阶段，检查 DLL 中是否存在 IsUnsafe 标记或特征。一旦包含 unsafe 代码块，直接在加载阶段抛出异常并终止加载。
  - 报错反馈：触发 UI 全局弹窗："您安装的 Mod [名称] 企图调用反射或使用了 unsafe 非托管代码，已被安全系统拦截。建议移除该 Mod。"
- GDScript：
  - Godot GDScript 本身不具备反射和 unsafe 能力。仅需在底层劫持 GDScript::call() 的虚函数表。若 call() 企图指向 Engine 或 OS 的私有方法，直接抛回 null 并记录日志。

### 2. DllImport 与外部命令执行

- C# 拦截：ModAssemblyLoader 在加载 DLL 时，深度扫描元数据。如果发现 `[DllImport]` 特性，直接拒绝加载或将该标记方法封存，阻断 Mod 直接调用 Windows 原生 API 的能力。
- GDScript 拦截：对 `OS.execute` 等底层系统调用进行底层 Hook。一旦检测到，底层引擎返回 false（进程启动失败），沙箱日志进行记录。

### 3. 网络访问隔离 (Network Isolation)

- 拦截 HTTPRequest 节点的底层网络包收发（通过 Godot 的 HTTPClient 拦截）。
- 严格/绝对严格模式：直接屏蔽 `HTTPRequest.request()` 的执行，使其永远处于失败状态。
- 相对严格模式：限制单 Mod 60 秒内上行流量不超过 64KB。下行数据不设限。

## 四、 事件监控、防死锁与 UI 交互

### 1. 绝对的加载顺序（防止沙箱自身被劫持）

- 架构调整：沙箱主类修改为同步创建，移除所有 call_deferred。必须确保在 Main.tscn 的 `_Ready()` 执行完毕、Mod 管理器初始化之前，底层 API Hook 已经完全植入。

### 2. 异步权限请求与模态屏阻断（防死锁）

- 底层拦截器发现违规操作时，必须主动挂起当前 Mod 的执行线程（或状态机）。
- UI 回调：调用 `ModAPI.ShowPermissionDialog(string modId, string msg, Action onAllow, Action onDeny)`。
- 在玩家点击"允许/拒绝"前，沙箱保持挂起状态。玩家确认后，根据选择决定返回文件句柄权限或抛出拒绝异常，恢复 Mod 执行。

---

## 五、 透明监控与日志系统 (Logging)

- 日志存储位置：`user://mods_security_logs/`（绝对不能被 Mod 访问；底层拦截任何企图读取此目录的请求）。
- 开放模式：记录所有 Mod 的高危操作与拦截轨迹（后期可由作者查看排查问题）。
- 绝对开放模式：不写入任何日志。玩家选择此模式即代表承担一切风险，沙箱系统不作任何限制与留痕（符合原案设计）。

---

## 六、 备注与底层技术栈（给开发者的提示）

- 这套方案的落地关键在于 "原生层函数表劫持"。根据 Godot 4 的架构，可以通过 C# 的 Godot.NativeInterop 获取 godot_double 和 godot_string 的底层函数表（VTables）进行替换。
- 无需大改现有的 ModManager 和 ModAPI 的调度流程，只需将 "沙箱校验逻辑" 直接挂载在 Godot 原生 C++ 到 C# 的 NativeCall 拦截层之上即可。
- 针对 unsafe 的说明：因为 unsafe 允许 Mod 直接从托管代码（C#）跳跃到非托管 C++ 内存，一旦绕过，沙箱的 API 劫持就会失效。因此，针对 unsafe 在元数据加载阶段的预阻断，是本沙箱系统防止 C# Mod 将安全风险降至与 GDScript 同等水平的最核心防线。
