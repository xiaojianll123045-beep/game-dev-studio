# 平衡调整

数据模组最常见的用途是调整游戏平衡性。通过 `balance.json` 文件，你可以覆盖游戏中的各种浮点数值。

## 文件位置

文件名**必须**为 `balance.json`，放在 `data/` 目录下：

```
data/
└── balance.json
```

## 基础格式

`balance.json` 是一个扁平的 **key → float 映射**。代码会逐条解析，将每条键值对写入覆盖表。

```json
{
  "project.init.graphics": 40,
  "project.init.stability": 60,
  "fatigue.monthly_gain": 2,
  "hire.cost.level3": 250000
}
```

## 所有可覆盖的平衡键

### 项目初始属性 (project.init)

新项目创建时的各项基础评分：

| Key | 默认值 | 说明 |
|-----|--------|------|
| `project.init.graphics` | 30 | 美术基础分 |
| `project.init.gameplay_base` | 25 | 玩法基础分 |
| `project.init.gameplay_compat` | 15 | 玩法 × 题材兼容性加成 |
| `project.init.audio` | 20 | 音乐基础分 |
| `project.init.network` | 10 | 网络基础分 |
| `project.init.story` | 15 | 剧情基础分 |
| `project.init.stability` | 50 | 稳定性基础分（Bug 相关） |

### 评分权重 (score.weight)

最终评分中各维度的权重系数：

| Key | 默认值 | 说明 |
|-----|--------|------|
| `score.weight.graphics` | 0.20 | 美术权重 |
| `score.weight.gameplay` | 0.30 | 玩法权重 |
| `score.weight.audio` | 0.10 | 音乐权重 |
| `score.weight.story` | 0.15 | 剧情权重 |
| `score.weight.network` | 0.10 | 网络权重 |
| `score.weight.stability` | 0.15 | 稳定性权重 |
| `score.weight.scale_factor` | 0.20 | 规模因子权重 |
| `score.weight.scale_base` | 0.90 | 规模基准值 |

### 销量计算 (sales)

游戏发售后销量计算相关：

| Key | 默认值 | 说明 |
|-----|--------|------|
| `sales.multiplier.95` | 1.5 | 评分 ≥ 95 的销量倍率 |
| `sales.multiplier.90` | 1.2 | 评分 ≥ 90 |
| `sales.multiplier.80` | 1.0 | 评分 ≥ 80 |
| `sales.multiplier.70` | 0.8 | 评分 ≥ 70 |
| `sales.multiplier.60` | 0.6 | 评分 ≥ 60 |
| `sales.multiplier.below60` | 0.3 | 评分 < 60 |
| `sales.base_constant` | 3000 | 基础销量常数 |
| `sales.base_per_score` | 350 | 每评分点销量 |

### 招聘费用 (hire.cost)

按员工等级划分的外部招聘价格：

| Key | 默认值 | 说明 |
|-----|--------|------|
| `hire.cost.level0` | 5,000 | 初级员工 |
| `hire.cost.level1` | 20,000 | 中级员工 |
| `hire.cost.level2` | 80,000 | 高级员工 |
| `hire.cost.level3` | 300,000 | 资深员工 |
| `hire.cost.level4` | 800,000 | 顶级员工 |

### 员工疲劳 (fatigue)

| Key | 默认值 | 说明 |
|-----|--------|------|
| `fatigue.monthly_gain` | 3 | 员工每月疲劳增长值 |

### 特质修饰 (trait)

每个员工特质可覆盖效率、疲劳增速、满意度衰减系数。以特质名全小写为 key：

```json
{
  "trait.workaholic.efficiency": 1.1,
  "trait.workaholic.fatigue": 0.5,
  "trait.chill.efficiency": 0.9,
  "trait.chill.fatigue": 0.7,
  "trait.chill.satisfaction_decay": 0.3,
  "trait.perfectionist.efficiency": 1.05,
  "trait.lone_wolf.efficiency": 1.2,
  "trait.ambitious.fatigue": 1.2,
  "trait.ambitious.satisfaction_decay": 2.0,
  "trait.sensitive.satisfaction_decay": 1.8,
  "trait.lucky.satisfaction_decay": 0.5
}
```

## 完整示例

```json
{
  "project.init.graphics": 35,
  "project.init.stability": 55,
  "score.weight.stability": 0.2,
  "sales.multiplier.80": 1.2,
  "hire.cost.level0": 8000,
  "fatigue.monthly_gain": 2
}
```

## 数据合并规则

多个数据模组的 `balance.json` 会**合并**到同一个覆盖表中：

1. 每条键值对独立覆盖
2. 后加载的模组中存在的 key 会覆盖先加载的值
3. 不存在的 key 保留先前的值（不受影响）

## 注意事项

- 数值必须是浮点数或整数（会自动转为 float）
- 不认识的 key 会直接加入覆盖表，但不会被游戏读取
- 修改前请记录原始数值，方便恢复
- 平衡性修改建议分步进行，多次测试
